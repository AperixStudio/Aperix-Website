Recommended GitHub Org Layout

`AperixStudio`

AperixStudio/ ├─ Aperix-Website │ ├─ public agency website │ ├─ demos / portfolio pieces │ └─ lead capture / enquiry flow │ ├─ aperix-admin │ ├─ internal dashboard │ ├─ client records │ ├─ health checks │ └─ hosting / repo / access links │ ├─ aperix-site-starter │ ├─ reusable Next.js starter │ ├─ shared blocks and patterns │ └─ launch baseline for new clients │ ├─ client-rhinos-walk-site ├─ client-future-cafe-site ├─ client-future-tradie-site │ └─ aperix-internal ├─ SOPs ├─ docs └─ small scripts / operations

What each repo is for

`Aperix-Website`

Public marketing site, live demos, portfolio, pricing, and contact funnel.

`aperix-admin`

Private command center for both of you to manage operational reality.

`aperix-site-starter`

Starter base for new builds so you stop rebuilding everything from zero.

Client repo pattern

Example: `client-rhinos-walk-site`

client-rhinos-walk-site/ ├─ app/ ├─ components/ ├─ lib/ ├─ public/ ├─ docs/ │ ├─ content-notes │ ├─ launch-checklist │ └─ handoff-notes ├─ .env.example ├─ netlify.toml └─ README (optional internal use)

How work flows

Lead enters through `Aperix-Website` Contact form captures the enquiry and later feeds a client/project record into the admin app.
Project is accepted Create a new client repo from `aperix-site-starter` and assign lead/support owners.
Site gets hosted Connect GitHub → Netlify → Cloudflare → domain registrar and document links in admin.
Admin app becomes the control layer It does not replace the repos — it sits above them and gives you one operational view.
